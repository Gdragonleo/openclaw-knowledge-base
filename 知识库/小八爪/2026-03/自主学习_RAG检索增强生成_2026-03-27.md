## 一、RAG是什么？

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种混合框架，将信息检索机制与生成式模型结合，提升AI回答的上下文相关性和事实准确性。

### 核心解决的问题

| 问题 | 说明 | RAG如何解决 |
|------|------|------------|
| **知识时效性不足** | LLM训练数据固定，无法动态更新 | 检索实时外部知识 |
| **幻觉（Hallucination）** | 生成模型经常编造看似合理但错误的信息 | 用检索到的事实约束生成 |
| **可扩展性** | 模型记忆有限 | 绕过记忆限制，访问海量外部数据库 |

### 核心思想（一句话）
> 在生成答案之前，先从知识库中检索相关信息，用检索到的事实来指导生成过程。

## 二、RAG架构详解

### 2.1 三大核心组件

```
用户提问
    ↓
┌─────────────────┐
│  Embedding Model │  → 将查询和文档转为向量
└────────┬────────┘
         ↓
┌─────────────────┐
│    Retriever     │  → 用相似度匹配，找最相关的文档
└────────┬────────┘
         ↓
┌─────────────────┐
│    Generator     │  → LLM基于（查询+检索文档）生成回答
└────────┬────────┘
         ↓
    最终回答
```

| 组件 | 功能 | 常用工具 |
|------|------|---------|
| **Embedding Model** | 将文本转为稠密向量 | OpenAI Ada、Sentence-BERT |
| **Retriever** | 用余弦相似度找top-k文档 | Pinecone、Weaviate、Qdrant、FAISS |
| **Generator** | 基于上下文生成回答 | GPT、GLM、Claude等 |

### 2.2 四步工作流程

1. **Query Encoding（查询编码）**
   - 用户输入通过Embedding模型转为向量
   
2. **Document Retrieval（文档检索）**
   - 在向量数据库中用ANN（近似最近邻）搜索
   - 返回与查询最相似的top-k文档
   
3. **Contextual Fusion（上下文融合）**
   - 检索到的文档附加到查询后面
   - 可能做摘要/截断以适应模型输入长度限制
   
4. **Response Generation（响应生成）**
   - 生成模型基于（查询+检索文档）产生回答

## 三、传统RAG的局限性

### 3.1 检索质量问题
- **相关性不足**：检索系统返回的文档可能不符合查询意图
- **知识库缺陷**：不完整或过时的知识库导致信息缺口

### 3.2 上下文理解不足
- **查询歧义**：模糊或表述不当的查询导致检索偏差
- **多跳推理**：无法跨多个检索文档连接信息

### 3.3 仍然存在幻觉
- 生成模型可能曲解检索到的文档内容
- 即使提供了正确文档，也可能生成错误回答

### 3.4 高延迟
- 传统RAG将文档切成约100词的小块
- 搜索空间巨大（数百万个小块），增加延迟

## 四、高级RAG技术（2025前沿）

### 4.1 Long RAG（长文本RAG）

**核心改进**：不再切小块，而是处理更长的文档单元（整节或整篇）

| 对比 | 传统RAG | Long RAG |
|------|---------|----------|
| 文档单元 | ~100词小块 | 整节/整篇文档 |
| 上下文 | 易碎片化 | 保持完整上下文 |
| 搜索空间 | 数百万小块 | 大幅减少 |
| 延迟 | 较高 | 显著降低 |

**适用场景**：法律文档、学术论文、技术手册、企业知识库

### 4.2 Self-RAG（自反思RAG）

**核心改进**：引入自我反思机制，动态决定何时检索、评估检索结果质量

**四大能力**：
1. **自适应检索**：只在需要时才检索，不浪费算力
2. **选择性引用**：评估检索文档的相关性和证据支持度
3. **反思Token**：用特殊标记引导决策（Retrieve/ISREL/ISUSE）
4. **批判机制**：生成后自我评估，迭代优化

**关键区别**：
- 传统RAG：盲目检索固定数量文档 → 生成
- Self-RAG：判断是否需要检索 → 选择性检索 → 评估相关性 → 自我批评 → 优化输出

### 4.3 Graph RAG（图增强RAG）

**核心改进**：利用知识图谱建立文档间的结构关系

- 将文档转为知识图谱（实体+关系）
- 支持多跳推理（跨文档关联）
- 适合需要深层关联分析的场景

### 4.4 Agentic RAG（智能体RAG）

**核心改进**：用AI Agent替代固定检索流程

- Agent可以自主决定检索策略
- 支持多工具调用（搜索、数据库查询、API调用）
- 可以多轮检索优化结果
- 2025-2026最热门方向

## 五、向量数据库选型

| 数据库 | 类型 | 特点 | 适用场景 |
|--------|------|------|---------|
| **FAISS** | 开源/本地 | Meta开源，Python友好 | 开发测试、小规模 |
| **ChromaDB** | 开源/本地 | 轻量，API简洁 | 快速原型、个人项目 |
| **Qdrant** | 开源/可部署 | Rust编写，高性能 | 中大规模生产 |
| **Weaviate** | 开源/云端 | 内置向量化 | 需要一站式方案 |
| **Pinecone** | 云服务 | 全托管，零运维 | 企业级生产 |
| **Milvus** | 开源/可扩展 | 中国团队，生态好 | 大规模企业 |

**小八爪选型建议**：
- 本地轻量：**ChromaDB**（最简单上手）
- 中等规模：**Qdrant**（性能好）
- 国内部署：**Milvus**（中国生态，文档中文友好）
