# [2026-03-10] 定时任务执行但未实际发送通知

**错误类型**：任务执行
**严重程度**：🔴 高（影响所有定时任务）

## 问题现象

小刘反馈：
> "今天早上的汇报你没有进行，我没看见每日国际新闻的汇总，然后每日招标报告、文献研究员的自主学习报告都没有，还有daily literature research都没有发给我"

检查发现：
- ✅ 定时任务状态显示"ok"（执行成功）
- ✅ 任务执行时间正确（9:00、2:00、22:00等）
- ❌ deliveryStatus：**not-requested**（没有发送通知）
- ❌ 用户没有收到任何通知
- ❌ 任务只是触发了，但没有实际执行

**受影响的任务**：
1. 每日招标超详细报告（9:00）
2. 每日国际新闻汇总（9:00）
3. 每日工作学习汇报（9:00）
4. 文献研究员自主学习（22:00）
5. Daily literature research（2:00，还有error）

## 根本原因

1. **HEARTBEAT.md配置问题**
   - HEARTBEAT.md只提供了任务清单
   - **没有提供具体的执行指令**
   - 定时任务触发后，不知道要做什么

2. **任务执行机制**
   - 定时任务触发 → 读取HEARTBEAT.md
   - HEARTBEAT.md只有任务描述
   - **没有具体的执行步骤和指令**
   - 任务状态显示"ok"，但实际没有执行工作

3. **缺乏验证机制**
   - 没有检查deliveryStatus
   - 没有确认用户是否收到通知
   - 没有验证任务是否真正完成

## 影响范围

- 🔴 所有依赖HEARTBEAT.md的定时任务
- 🔴 用户期望的自动化汇报全部失败
- 🔴 用户信任严重受损
- 🔴 定时任务系统形同虚设

## 解决方案

### 已执行：
1. ✅ 立即补发所有缺失的报告
   - 智能监测技术文献调研报告（已发送）
   - 国际新闻汇总（子智能体执行中）
   - 招标报告（子智能体执行中）

2. ✅ 记录错误到error-lessons

### 长期方案：
1. 修改HEARTBEAT.md，添加具体的执行指令
2. 每个定时任务配置文件包含完整的执行步骤
3. 建立任务执行验证机制

## 预防规则

### HEARTBEAT.md改进：

```markdown
## 任务X：XXX（触发时间）

**触发条件**：XXX

**任务内容**：
1. 具体步骤1
2. 具体步骤2
3. 具体步骤3

**执行指令**：
- 使用 sessions_spawn 创建子智能体
- 任务描述：[详细描述]
- 模型：zai/glm-5
- 输出：[明确位置]
- 通知：[发送给小刘]

**验证步骤**：
1. 检查产出文件是否生成
2. 检查飞书通知是否发送
3. 主动向小刘确认收到
```

### 定时任务配置文件改进：

每个cron_tasks/下的配置文件必须包含：

```markdown
## 任务执行指令

**创建子智能体**：
```
sessions_spawn({
  task: "具体任务描述",
  model: "zai/glm-5",
  timeoutSeconds: 600,
  label: "任务标签"
})
```

**产出验证**：
- 检查文件：`/path/to/output`
- 检查通知：飞书是否发送
- 确认收到：主动询问小刘
```

### 验证机制：

**每个定时任务执行后，必须检查**：

```bash
# 1. 检查任务执行状态
openclaw cron runs --id <job_id> --limit 1

# 2. 验证deliveryStatus
# 必须不是"not-requested"或"unknown"

# 3. 验证产出文件
ls -lh /path/to/output

# 4. 验证飞书通知
# 查看是否发送成功
```

### 任务执行后主动验证：

```markdown
重要定时任务执行后（9:00的任务）：

1. 执行任务（sessions_spawn）
2. 等待完成
3. 检查产出文件是否存在
4. 检查飞书通知是否发送
5. 如果deliveryStatus是"not-requested"：
   - 立即手动发送通知
   - 记录错误
   - 修复配置
```
