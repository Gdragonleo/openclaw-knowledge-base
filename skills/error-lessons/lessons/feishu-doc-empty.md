# 教训7：飞书文档创建失败 - 内容丢失

**错误类型**：文档创建
**严重程度**：🔴 高（影响用户体验）
**创建时间**：2026-03-13 00:22

---

## 问题场景

使用 `feishu_doc` 工具创建文档时，只创建了标题，内容为空。

---

## 错误现象

```json
// 调用成功，但内容丢失
{
  "action": "create",
  "content": "# 标题\n\n很长的详细内容...",
  "结果": "只创建了标题，内容为空"
}
```

**实际结果**：
```
标题：📋 招标系统定制化信息清单（10个方面）
内容：空白 ❌
```

---

## 根本原因

### **可能原因**：
1. ❌ Content 参数过长（超过限制）
2. ❌ Markdown 格式与飞书不兼容
3. ❌ API 调用部分失败（没有明确报错）
4. ❌ 特殊字符或格式导致内容丢失

### **实际原因**：
- **Content 过长**：约4000行，超过飞书单次创建限制
- **Markdown 复杂**：表格、代码块、列表嵌套过多

---

## 影响范围

- 🔴 用户打开文档看到空白
- 🔴 失去信任
- 🔴 需要重新创建
- 🔴 浪费时间

---

## 解决方案

### **方案1：创建本地 Markdown 文件** ⭐⭐⭐⭐⭐
```bash
✅ 创建完整的 .md 文件
✅ 提交到 GitHub
✅ 提供 GitHub 链接
✅ 稳定可靠
```

### **方案2：分段创建飞书文档** ⭐⭐⭐⭐
```bash
✅ 分10个文档（每个方面一个）
✅ 内容简短，不易丢失
✅ 创建索引文档链接
```

### **方案3：使用飞书文档 API（分段）** ⭐⭐⭐
```bash
✅ 先创建空文档
✅ 逐段 append 内容
✅ 更可靠
```

---

## 预防规则

### **创建飞书文档时**：
```
1. ✅ 先判断内容长度
   - < 1000 行：直接创建
   - > 1000 行：分段创建或用本地文件

2. ✅ 创建后立即验证
   - 读取文档内容
   - 确认非空
   - 确认内容完整

3. ✅ 优先使用本地文件
   - GitHub 链接更稳定
   - Markdown 格式兼容
   - 版本控制

4. ✅ 提供备选方案
   - 同时提供 GitHub 链接
   - 避免单点失败
```
