## 一、多档案支持算法设计

### 1.1 数据结构设计

#### 1.1.1 用户档案表 (UserProfiles)

```json
{
  "profile_id": "string (UUID)",
  "user_id": "string (用户唯一标识)",
  "profile_name": "string (档案名称)",
  "profile_type": "enum {SELF, FAMILY, FRIEND}",
  "created_at": "timestamp (创建时间)",
  "updated_at": "timestamp (更新时间)",
  "is_active": "boolean (是否激活)"
}
```

#### 1.1.2 档案敏感信息表 (ProfileSensitiveData)

```json
{
  "profile_id": "string (UUID, 关联 UserProfiles)",
  "encrypted_birth_datetime": "string (加密后的出生时间)",
  "encrypted_birth_location": "string (加密后的出生地点)",
  "encryption_version": "string (加密算法版本)",
  "encryption_key_id": "string (密钥ID)"
}
```

**加密方案说明：**
- 使用 AES-256-GCM 对称加密
- 每个用户独立的加密密钥（派生自用户主密钥 + profile_id）
- 密钥存储在密钥管理服务（KMS）中，不在应用层明文存储
- 加密流程：
  1. 从 KMS 获取用户主密钥
  2. 使用 HKDF 派生档案专属密钥：`profile_key = HKDF(master_key, profile_id)`
  3. 使用 AES-256-GCM 加密敏感数据
  4. 存储密文 + nonce + auth_tag

#### 1.1.3 排盘结果表 (ChartingResults)

```json
{
  "result_id": "string (UUID)",
  "profile_id": "string (UUID, 关联 UserProfiles)",
  "charting_type": "string (排盘类型：八字、紫微、六爻等)",
  "charting_data": "object (排盘数据JSON)",
  "metadata": {
    "calculated_at": "timestamp (计算时间)",
    "algorithm_version": "string (算法版本)"
  }
}
```

#### 1.1.4 历史查询记录表 (QueryHistory)

```json
{
  "query_id": "string (UUID)",
  "profile_id": "string (UUID, 关联 UserProfiles)",
  "query_type": "string (查询类型)",
  "query_params": "object (查询参数)",
  "response_summary": "string (响应摘要)",
  "queried_at": "timestamp (查询时间)"
}
```

### 1.2 核心算法逻辑

#### 1.2.1 创建新档案

```
输入: user_id, profile_name, profile_type, birth_datetime, birth_location
输出: profile_id

算法步骤:
1. 生成唯一 profile_id = UUIDv4()
2. 验证 profile_name 唯一性
   - 查询 UserProfiles WHERE user_id AND profile_name
   - 若存在 → 返回错误
3. 创建 UserProfiles 记录记录
4. 加密敏感数据
   - profile_key = HKDF(master_key, profile_id)
   - encrypted_birth = AES-GCM(profile_key, birth_datetime)
   - encrypted_location = AES-GCM(profile_key, birth_location)
5. 创建 ProfileSensitiveData 记录
6. 触发首次排盘计算
7. 返回 profile_id

边界条件:
- 档案名称重复 → 拒绝
- 用户未创建加密密钥 → 提示先设置安全口令
- birth_datetime 格式无效 → 拒绝

时间复杂度: O(1) (假设 UUID 生成和加密为常数时间)
空间复杂度: O(1)
```

#### 1.2.2 读取档案信息（解密）

```
输入: profile_id, user_id
输出: profile_data (含解密后的敏感信息)

算法步骤:
1. 验证档案归属
   - 查询 UserProfiles WHERE profile_id AND user_id
   - 若不存在或 user_id 不匹配 → 返回权限错误
2. 获取 ProfileSensitiveData 记录
3. 获取密钥
   - 从 KMS 获取用户主密钥
   - profile_key = HKDF(master_key, profile_id)
4. 解密敏感数据
   - birth_datetime = AES-GCM-Decrypt(profile_key, encrypted_birth)
   - birth_location = AES-GCM-Decrypt(profile_key, encrypted_location)
5. 组装完整档案数据返回

边界条件:
- 档案不存在 → 404
- 用户无权限 → 403
- 解密失败（密钥错误、数据损坏）→ 返回数据损坏错误

时间复杂度: O(1)
空间复杂度: O(1)
```

#### 1.2.3 批量查询用户档案列表

```
输入: user_id
输出: profiles[] (不含敏感信息)

算法步骤:
1. 查询 UserProfiles WHERE user_id AND is_active = true
2. 映射为简化的档案对象
   - 包含: profile_id, profile_name, profile_type, created_at
   - 排除: 敏感数据、详细排盘结果
3. 按更新时间倒序排列

边界条件:
- 用户无档案 → 返回空数组

时间复杂度: O(n), n 为用户档案数量
空间复杂度: O(n)
```

### 1.3 加密方案详细设计

#### 1.3.1 密钥派生方案

```
使用 HKDF (RFC 5869) 进行密钥派生：

输入:
- master_key: 从 KMS 获取的用户主密钥 (256-bit)
- salt: 固定值 "liushi_profile_salt"
- info: profile_id (UTF-8 编码)
- output_length: 32 bytes (AES-256 密钥长度)

算法:
profile_key = HKDF-Extract(master_key, salt)
profile_key = HKDF-Expand(profile_key, info, 32)

优势:
- 每个档案派生独立密钥
- 主密钥泄露不会直接泄露所有档案数据
- 可通过重新派生密钥支持密钥轮换
```

#### 1.3.2 加密格式

```
加密后的数据存储格式（Base64 编码）:

{
  "ciphertext": "string (AES-GCM 密文)",
  "nonce": "string (96-bit nonce, Base64)",
  "auth_tag": "string (128-bit authentication tag, Base64)"
}

解密验证:
1. 解析 Base64 各字段
2. 使用 AES-256-GCM 解密，验证 auth_tag
3. 若验证失败 → 拒绝解密
```

## 二、每日推送逻辑算法设计

### 2.1 数据结构设计

#### 2.1.1 推送任务表 (PushTasks)

```json
{
  "task_id": "string (UUID)",
  "user_id": "string (用户ID)",
  "profile_id": "string (档案ID)",
  "scheduled_date": "date (YYYY-MM-DD)",
  "scheduled_time": "time (用户当地时间 HH:MM)",
  "timezone": "string (时区，如 Asia/Shanghai)",
  "status": "enum {PENDING, SENT, SKIPPED, FAILED}",
  "sent_at": "timestamp (实际发送时间)",
  "skip_reason": "string (跳过原因，若适用)"
}
```

#### 2.1.2 推送缓存表 (PushCache)

```json
{
  "cache_id": "string (UUID)",
  "profile_id": "string (档案ID)",
  "date": "date (YYYY-MM-DD)",
  "content_hash": "string (推送内容 SHA256)",
  "significance_score": "number (重要性评分 0-100)",
  "cached_at": "timestamp (缓存时间)"
}
```

#### 2.1.3 用户推送偏好表 (UserPushPreferences)

```json
{
  "user_id": "string",
  "profile_id": "string",
  "push_enabled": "boolean (是否启用推送)",
  "push_time": "time (期望推送时间，默认 08:00)",
  "timezone": "string ( (时区)",
  "min_significance_threshold": "number (最小重要性阈值，默认 30)"
}
```

### 2.2 核心算法逻辑

#### 2.2.1 每日调度器（Cron Job）

```
触发频率: 每 5 分钟
执行逻辑:

FOR EACH user_id in 活跃用户列表:
  1. 获取用户时区
  2. 计算用户当前时间
  3. IF 当前时间 >= 08:00 且 当前时间 < 08:05:
       触发推送评估 (user_id, 当前日期)

边界条件:
- 用户时区无效 → 使用默认 Asia/Shanghai
- 用户无推送偏好 → 跳过

时间复杂度: O(U), U 为活跃用户数
空间复杂度: O(1)
```

#### 2.2.2 推送评估算法（核心）

```
输入: user_id, date (YYYY-MM-DD)
输出: 决策 (发送/跳过)

算法步骤:

1. 获取用户所有激活档案
   profiles = QueryUserActiveProfiles(user_id)

2. FOR EACH profile in profiles:
   a. 检查今日是否已推送
      task = QueryPushTask(profile_id, date)
      IF task.status == SENT OR SKIPPED:
         CONTINUE

   b. 计算今日运势
      fortune_data = CalculateDailyFortune(profile_id, date)

   c. 提取关键信息
      - 时辰吉凶数组 [吉, 吉, 凶, 凶, 凶, 吉, 凶, 凶, 吉, 凶, 凶, 吉]
      - 当日宜忌列表
      - 三合/冲煞信息

   d. 计算重要性评分
      score = CalculateSignificanceScore(fortune_data)

   e. 节流判断
      cache = QueryPushCache(profile_id, date)
      IF cache 存在:
         IF cache.content_hash == 当前内容哈希:
            score -= 50 (大幅降权)
         IF cache.significance_score >= score:
            跳过推送

   f. 用户偏好判断
      preferences = QueryPushPreferences(user_id, profile_id)
      IF NOT preferences.push_enabled:
         标记为跳过，原因: 用户关闭推送
      IF score < preferences.min_significance_threshold:
         标记为跳过，原因: 重要性不足

   g. 决策
      IF score >= 阈值:
         生成推送消息
         发送推送 (user_id, message)
         更新 PushTasks 状态为 SENT
         记录推送日志
      ELSE:
         更新 PushTasks 状态为 SKIPPED
         记录为 SKIPPED

3. 返回处理统计

边界条件:
- 档案排盘数据缺失 → 跳过
- 运势计算失败 → 标记为 FAILED
- 推送服务不可用 → 标记为 FAILED，支持重试
```

#### 2.2.3 重要性评分算法

```
输入: fortune_data
输出: score (0-100)

计算逻辑:

score = 0

# 1. 时辰吉凶权重 (30分)
lucky_hours = Count时辰吉(fortune_data.时辰吉凶)
score += (lucky_hours / 12) * 30

# 2. 冲煞权重 (40分)
IF fortune_data.冲煞信息.与日主相冲:
   score += 40
ELSE IF fortune_data.冲煞信息.与日主相害:
   score += 25
ELSE IF fortune_data.冲煞信息.与日主相刑:
   score += 15
ELSE IF fortune_data.冲煞信息.三合:
   score += 35

# 3. 宜忌数量权重 (20分)
yi_count = Count(fortune_data.当日宜)
ji_count = Count(fortune_data.当日忌)
score += (yi_count + ji_count) * 2
IF score > 20: score = 20

# 4. 特殊日历权重 (10分)
IF fortune_data.是节日 OR 重大节气:
   score += 10

# 5. 历史变化对比 (额外加分)
cache = QueryPushCache(profile_id, date-1)
IF cache 存在 AND 缓存内容与今日差异大:
   score += 10

RETURN min(score, 100)
```

#### 2.2.4 推送消息生成算法

```
输入: fortune_data, profile_name
输出: message (结构化消息)

消息模板:

📅 {date} · {profile_name} 每日运势

⏰ 时辰吉凶
{列出吉时和凶时}

✅ 今日宜
{宜事列表}

❌ 今日忌
{忌事列表}

⚡ 冲煞提醒
{冲煞详情（若无则省略）}

💡 建议事项
{基于运势生成个性化建议}

---

优化点:
- 若所有字段都无显著变化，返回精简版
- 只显示有变化的区块
- 支持用户自定义显示偏好
```

### 2.3 节流优化策略

#### 2.3.1 内容去重

```
算法:
1. 计算推送内容的 SHA256 哈希
2. 与昨日缓存对比
3. 若哈希完全相同:
   - 检查重要性评分
   - 若评分 < 阈值 → 跳过推送
   - 若评分 >= 阈值 → 降低优先级，但保留推送

效果:
- 避免连续多日推送相同内容
- 保留重要事件的提醒（如连续冲煞）
```

#### 2.3.2 时区感知调度

```
算法:
1. 每个用户存储 preferred_timezone
2. 全局调度器每 5 分钟执行一次
3. 计算当前 UTC 时间
4. 遍历所有时区
   FOR each timezone in supported_timezones:
     local_time = UTC + timezone_offset
     IF 08:00 <= local_time < 08:05:
        users_in_timezone = QueryUsersByTimezone(timezone)
        FOR user in users_in_timezone:
           TriggerPushEvaluation(user, today)

优势:
- 支持全球用户
- 每个用户都在其当地时间 8:00 收到推送
- 避免高峰期推送拥堵（时区分散）
```

#### 2.3.3 重试与失败处理

```
策略:
1. 推送失败 → 标记为 FAILED
2. 重试次数: 最多 3 次
3. 重试间隔: 指数退避 (5min, 15min, 60min)
4. 超过重试次数 → 标记为 FAILED，记录错误日志
5. 失败统计 → 若连续失败 > 7 天，暂停该用户推送并通知

监控指标:
- 推送成功率
- 平均发送延迟
- 失败原因分布
```

## 三、边界条件与异常处理

### 3.1 多档案模块

| 场景 | 处理策略 |
|------|---------|
| 用户创建超过 50 个档案 | 拒绝创建，提示升级账户 |
| 档案名称包含特殊字符 | 过滤或拒绝 |
| 出生时间不在有效范围 | 拒绝（限制：1900-2100年） |
| 地理位置无法解析 | 使用默认值或提示用户 |
| 加密密钥丢失 | 标记档案为"不可访问"，提供恢复选项 |
| 并发创建同名档案 | 使用分布式锁，原子性检查-创建 |
| 删除激活中的档案 | 软删除，保留历史数据 30 天 |

### 3.2 每日推送模块

| 场景 | 处理策略 |
|------|---------|
| 用户时区变更 | 计算已发送/待发送的日期，补发或跳过 |
| 用户关闭推送 | 立即停止调度，清空待发送任务 |
| 推送服务限流 | 分批发送，按优先级排序 |
| 用户设备离线 | 记录失败，下次上线时重试 |
| 跨时区旅行 | 检测时区变化，调整下次推送时间 |
| 连续 7 天无变化 | 发送"运势平稳，暂无变化"汇总 |
| 农历闰月 | 使用标准农历库正确处理 |

### 3.3 性能边界

| 指标 | 目标值 | 监控阈值 |
|------|--------|---------|
| 单次排盘计算延迟 | < 500ms | > 1s 触发告警 |
| 单次推送评估延迟 | < 200ms | > 500ms 触发告警 |
| 并发推送吞吐量 | 1000 req/min | 拥塞时启用限流 |
| 数据库查询延迟 | < 50ms | > 100ms 触发告警 |
| 加密/解密延迟 | < 10ms | > 50ms 触发告警 |

## 四、复杂度分析

### 4.1 多档案模块

| 操作 | 时间复杂度 | 空间复杂度 | 说明 |
|------|-----------|-----------|------|
| 创建档案 | O(1) | O(1) | UUID 生成 + 加密为常数时间 |
| 读取档案（单个） | O(1) | O(1) | 索引查询 |
| 读取档案（批量） | O(n) | O(n) | n 为档案数量 |
| 更新档案 | O(1) | O(1) | 更新索引字段 |
| 删除档案（软删除） | O(1) | O(1) | 标记删除 |
| 加密/解密操作 | O(1) | O(1) | AES-256 固定长度数据 |

### 4.2 每日推送模块

| 操作 | 时间复杂度 | 空间复杂度 | 说明 |
|------|-----------|-----------|------|
| 单用户推送评估 | O(p) | O(p) | p 为用户档案数（通常 ≤ 10） |

| 内容去重检查 | O(1) | O(1) | 哈希对比 |

### 4.3 扩展性分析

**水平扩展能力:**
- 推送评估可并行化（按用户或时区分片）
- 数据库读写分离支持高并发
- 缓存层 (Redis) 降低数据库压力

**数据量预估:**
- 1 万用户 × 平均 5 档案 = 5 万档案记录
- 每日推送 = 1 万次（假设 20% 用户启用）
- 查询历史 = 每日 10 万条（假设每日 10 次查询）
- 年度数据增长 ≈ 3650 万条记录

**存储优化:**
- 查询历史表按月分片
- 超过 1 年的查询历史归档到冷存储
- 排盘结果使用对象存储 (S3)，数据库只存引用

## 五、安全与合规

### 5.1 数据安全

| 风险点 | 防护措施 |
|--------|---------|
| 敏感信息泄露 | AES-256-GCM 加密，密钥由 KMS 管理 |
| 密钥泄露风险 | 密钥派生使用 HKDF，主密钥不在应用层明文存储 |
| SQL 注入 | 使用参数化查询，ORM 框架防护 |
| 越权访问 | 档案归属验证，RBAC 权限控制 |
| 中间人攻击 | 全链路 HTTPS，证书固定 |

### 5.2 隐私合规

| 要求 | 实现方式 |
|------|---------|
| 数据最小化 | 只存储必要的出生时间和地点 |
| 用户控制权 | 提供档案导出、删除功能 |
| 数据留存策略 | 查询历史保留 1 年，超期自动清理 |
| 访问审计 | 记录所有敏感数据访问日志 |
| 合规声明 | 隐私政策明确说明数据处理方式 |

## 六、总结

### 6.1 核心设计亮点

1. **加密方案**
   - AES-256-GCM + HKDF 密钥派生
   - 档案级密钥隔离，主密钥安全存储

2. **推送节流**
   - 基于重要性评分的智能决策
   - 内容去重减少无效推送
   - 时区感知的分布式调度

3. **扩展性**
   - 数据结构支持水平扩展
   - 批处理支持高并发
   - 分片存储优化长期增长

### 6.2 后续优化方向

1. 引入机器学习模型优化重要性评分
2. 支持用户自定义推送模板
3. 添加推送效果分析（点击率、互动率）
4. 支持多语言运势输出
