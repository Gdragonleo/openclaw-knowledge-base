### 一、对比文件2（CN110096513A）与本发明"处理时机相反"的本质区别

#### 1.1 技术路径的根本差异

| 对比维度 | 对比文件2（CN110096513A） | 本发明 |
|---------|------------------------|--------|
| **核心技术** | 对已生成的SQL进行抽象语法树后置解析，用于标记关联阶段 | 在抽象语法树生成过程中动态注入安全约束，实现内生安全 |

#### 1.2 "先生成SQL，再后置解析"的技术局限

对比文件2采用的技术路径是：
1. 先根据自然语言生成SQL语句
2. 对已生成的SQL进行抽象语法树后置解析
3. 基于解析结果进行语法校验和关联阶段标记

**技术局限性**：
- SQL已经生成，即使校验失败，恶意SQL可能绕过校验机制
- 无法在SQL生成过程中实时干预数据边界
- 存在数据越权访问的潜在风险
- 采用"生成后校验"机制，安全检查属于事后补救

#### 1.3 "先构建语法树并注入约束，再生成SQL"的技术优势

本发明采用的技术路径是：
1. 根据三层架构经营逻辑知识图谱解构业务意图，生成逻辑算子
2. 在抽象语法树构建过程中，根据用户权限级别动态注入行级权限拦截算子
3. 将经过安全加固的抽象语法树转化为可执行的SQL语句
4. 执行SQL，获取数据

**技术优势**：
- 实现了"内生安全"，将安全约束内嵌至查询生成过程，而非事后补救
- 从源头杜绝了SQL注入和数据越权访问的可能性
- 即使攻击者构造恶意输入，也无法绕过权限边界
- 支持细粒度的权限控制（行级+列级），满足企业级数据安全管理需求
- 避免了后置校验的性能开销，提升了系统响应速度

#### 1.4 实质性区别结论

本发明与对比文件2在**处理时机**上存在根本性区别：
- 对比文件2：先生成SQL，再解析校验（后置）
- 本发明：先生成语法树并注入约束，再生成SQL（前置）

这种处理时机的差异导致两者在安全机制、安全保障、技术效果等方面存在本质区别。对比文件2无法给出本发明"在抽象语法树生成过程中动态注入安全约束"的技术启示。

具体体现为：
- 在抽象语法树构建过程中，根据用户权限级别动态注入行级权限拦截算子

- 从源头杜绝了不安全SQL的生成

#### 3.2 与后置校验的本质区别

| 对比维度 | 后置校验（对比文件2等） | 本发明的内生安全 |
|---------|------------------------|------------------|

### 四、归因分析是独创建造点（对比文件1/2/3均无）

#### 4.1 归因分析的技术定义

本发明的归因分析是指：
- 对查询结果进行时序分解和异常特征识别
- 采用Shapley值（源自合作博弈论）计算各业务因子（如业主类型、项目周期）对指标波动的贡献度
- 基于归因分析结果，生成诊断建议

归因分析实现从"数据呈现"到"智能诊断"的跨越，为经营者提供决策建议。

#### 4.2 归因分析的核心技术

**（1）时序分解**

采用STL（Seasonal-Trend decomposition using Loess）时序分解模型：
- 分解公式：Y(t) = Trend(t) + Seasonal(t) + Residual(t)
- Trend(t)：趋势分量，反映长期变化趋势
- Seasonal(t)：季节分量，反映周期性波动
- Residual(t)：残差分量，反映随机波动

技术效果：
- 识别指标波动的趋势性、季节性和异常性特征
- 为归因分析提供时序特征输入

**（2）Shapley值归因**

Shapley值源自合作博弈论，用于公平地分配各参与方对总收益收益的贡献。

计算公式：
```
φ_i = Σ [|S|! × (n - |S| - 1)! / n!] × [v(S ∪ {i}) - v(S)]
```

其中：
- φ_i：特征i的Shapley值（贡献度）
- S：特征子集（不包含特征i）
- n：特征总数
- v(S)：特征子集S对应的预测值
- v(S ∪ {i}) - v(S)：特征i的边际贡献

技术效果：
- 公平地计算各业务因子对指标波动的贡献度
- 提供可解释的归因分析结果

**（3）诊断建议生成**

基于归因分析结果，采用规则引擎生成诊断建议：
- 规则库设计：IF 条件 THEN 建议
- 建议生成流程：规则匹配 → 建议排序 → 建议整合

技术效果：
- 将归因分析结果转化为可操作的决策建议
- 提升经营决策的智能化水平

### 五、授权风险可控的综合论述

#### 5.1 本发明授权风险的全面保障

本发明通过多层次的安全机制，确保授权风险完全可控：

**（1）内生安全机制**

- 在抽象语法树生成过程中动态注入安全约束
- 从源头杜绝了不安全SQL的生成
- 杜绝了SQL注入和数据越权访问的可能性

**（2）细粒度权限控制**

- 行级权限：根据用户角色注入行级权限过滤条件（如"dept_code = 'DEPT_002'"）
- 列级权限：对SelectNode中的字段进行列级权限过滤，移除无权限字段或进行脱敏处理
- 动态权限注入：根据用户权限级别动态注入权限拦截算子

**（3）RBAC权限模型**

- 采用RBAC（基于角色的访问控制）模型
- 权限粒度：表级权限 + 行级权限 + 列级权限
- 权限配置表结构：
  ```
  user_role_table: (user_id, role_id)
  role_permission_table: (role_id, table_name, row_filter, column_list)
  ```

**（4）多层次安全校验**

1. 语法白名单校验：
   - 只允许SELECT查询，禁止INSERT/UPDATE/DELETE/DROP等修改操作
   - 禁止使用UNION、SUBQUERY等可能导致数据泄露的复杂语法
   - 禁止使用系统函数（如LOAD_FILE、INTO OUTFILE）

2. 参数化查询：
   - 所有用户输入的参数（时间、数值等）使用参数化查询，防止SQL注入
   - SQL模板中使用占位符（如"?"），参数单独传递

3. 执行计划分析：
   - 在执行前，使用EXPLAIN分析SQL执行计划
   - 检测全表扫描、笛卡尔积等低效操作，防止性能攻击
   - 若执行计划异常，拒绝执行并告警

4. 结果集数量限制：
   - 强制添加LIMIT子句，限制单次查询返回的最大记录数
   - 防止大量数据导出导致的信息泄露
